# 代码审查与优化建议(2026-07-06)

> 范围:后端 15 个 api + 12 个 lib + dev_server + scripts,前端 10 个 HTML + app.js/css(约 12000 行全量阅读)。
> 所有 P0 结论已在源码中逐条核实(附行号)。按 P0(安全/数据风险)→ P1(明显 bug/性能)→ P2(质量)排列。

---

## 总体结论

底子是健康的:参数化 SQL 全库无注入点、HMAC token 用了 `compare_digest`、角色实时查库、备份用 sqlite backup API、COS 私有桶 + 预签名 + 白名单、限速有双闸——这些都是对的,**不需要重构**。

但有三个系统性病灶:

1. **登录前门弱于后门**:token 本身很稳,但登录只靠三个可枚举字段、花名册公开、无限速、id6 未配即放行、停用用户可走 JSON 兜底复活——HMAC 的强度被前门架空。
2. **check 与 act 不同事务**:KOS 唯一组合分配、每人限额、每日配额全是"先查后写"分两个连接,ThreadingHTTPServer 并发下会撞车,且表上没有唯一约束兜底。
3. **"永不失败"哲学过度**:JSON 兜底、密钥回退链、哈希失败降级明文、`except: pass`——可用性兜底兜住了"数据说没有",停用/安全控制形同虚设。

## P0 — 安全/数据风险(建议本周内修)

### P0-1 前端 XSS 三连,叠加 token 存 localStorage = 账号可被完整窃取

| 位置 | 问题 |
|---|---|
| `public/index.html:1025/1034/1040` | `onclick='copy(${JSON.stringify(t)})'` — JSON.stringify 不转义单引号,而属性用单引号包裹。AI 文案含 `'` 即可突破属性边界注入 JS(已核实) |
| `public/index.html:911/926` | 品牌/产品名未 escapeHtml 直插 `<option>`,管理员录入的产品名可造成**对全员生效的存储型 XSS**(qa.html 同位置转义了,唯独 index 漏了;已核实) |
| `admin-users.html:135`、`admin-products.html:137/145` | `onclick="del(${id}, '${esc(name)}')"` — esc 把 `'` 转成 `&#39;`,但浏览器解析属性时先解码实体再执行 JS,`');...//` 即突破 |

token 和用户对象都在 localStorage(`app.js:13-21`),任何一处 XSS 都能外带 token,24h 内冒充任意用户含管理员。

**修复**(半天):全站消灭"动态数据进内联 onclick",改 `data-*` 属性 + 事件委托;index.html:911/926 补 escapeHtml;index.html:969/983 的 `data.error` 同样补转义。中期把 token 迁到 HttpOnly cookie。

### P0-2 登录可被枚举撞库

- `api/login.py:78-87`:GET /api/login **匿名**返回全公司部门+姓名花名册。
- `api/login.py` 无任何限速(已核实,ratelimit 只挂在 generate/qa/cover 上);无失败锁定。
- `id_last6` 未配置的用户直接放行(`lib/idhash.py:57-61` 返回 None → 不校验);工号多为连号可枚举。
- 所有响应 `Access-Control-Allow-Origin: *`,可跨域脚本化爆破。

**攻击链**:匿名拿花名册 → 枚举工号 → 对未配 id6 的用户(尤其管理员)直接拿到 24h token。管理员账号安全 = 三个可枚举字段。

**修复**:① 登录接口纳入限流(按 IP,分钟+每日双闸)+ 失败计数锁定;② 管理员强制 id6(或独立口令);③ 花名册接口要求先选部门按需返回或直接下线;④ CORS 收敛到自己域名。

### P0-3 停用/删除用户可"复活"登录

`lib/users_store.py:145-152`(已核实):`get_user` 在 **DB 正常查询但查不到人**时也无条件回退 users.json。后台把人 `active=0` 停用后,只要 TA 还在 users.json 里就能照常登录;JSON 里若标 `is_admin: true`,**停用的管理员权限照旧**。

**修复**(1 小时):区分"DB 异常"与"DB 无此人"——查询正常返回 None 就返回 None,仅连接/查询抛异常才走 JSON 兜底。

### P0-4 会话密钥回退链危险

`lib/session.py:29-46`:`SESSION_SECRET → COS_SECRET_KEY → DEEPSEEK_API_KEY`。且 `.env.example` **没列** SESSION_SECRET/ID6_SALT/KOS_IMG_SECRET,新部署必踩坑。后果:API key 泄露即可伪造任意管理员 token;轮换 COS 密钥(归档里本来就计划轮换!)会静默踢掉全站登录。`lib/idhash.py` 同链,后补 ID6_SALT 会让已存哈希全部失效、用户集体被锁。

**修复**(1 小时):删掉回退链,启动时无 SESSION_SECRET/ID6_SALT 直接拒绝启动(单机 systemd 没有配不了 env 的借口);`.env.example` 补齐全部 key。**注意:必须在轮换 COS 密钥之前先做这条**,否则轮换当天全员掉线。

### P0-5 错误信息裸奔回前端

约 11 个接口的兜底都是 `except Exception as e: self._json(500, {"error": str(e)})`。尤其 `api/generate.py:99`、`api/qa.py:287` 把 DeepSeek 上游返回体前 300 字拼进异常再回给客户端,可能带上游账号信息。

**修复**:统一 500 只回通用文案,detail 落服务端日志。建议借抽基类(P2-1)一次修完。

## P1 — 明显 bug / 数据一致性(建议下周修)

### P1-1 KOS 领取竞态:并发会发出重复组合 ⭐核心业务约束被破坏

`api/kos.py:257-286` + `lib/kos_store.py:279-334`:`pick_combo`(读)和 `record_pack`(写)是两个独立连接、独立事务,中间还夹着**多次 COS 下载 + 一次 LLM 调用**(秒级窗口)。`kos_packs` 表对 `(library_id, combo2/combo4)` 没有 UNIQUE 约束(已核实)。周四发任务全员同时领取,恰是最容易撞车的场景:两人拿到同一组素材 → "唯一不复用"被破坏、容量超发。`per_person` 限额检查同样是 check-then-act,双击可超领。

**修复**(半天):① 给 combo2/combo4 加唯一索引(先给历史数据去重);② 选组合+落库合并进单连接 `BEGIN IMMEDIATE` 事务,撞索引则重试;③ per_person 校验放进同一事务;④ COS 下载/出图挪到落库之后。

### P1-2 领取失败烧掉组合

`api/kos.py:286-295`:先 `record_pack` 占用组合,再本地拼图;拼图抛异常直接 500,组合永久作废、名额白占,容量(本就是 min(C(h,2),C(v,4)) 的稀缺资源)白白减少。**修复**:失败即删 pack,或与 P1-1 一并事务化回滚。

### P1-3 qa.py 失败日志永远为空

`api/qa.py:463`(已核实):`req if 'req' in dir() else {}` — 方法内 `dir()` 返回 self 属性而非局部变量,恒为 False,失败日志永远拿不到 brand/product/user 上下文。改 `locals()` 即可(1 分钟)。

### P1-4 admin.html 鉴权字段不一致(功能 bug)

`admin.html:569` 用 `App.roleOf()` 判权,`:1239` 启动却判旧字段 `me.is_admin` — `role='org_admin'` 但无 is_admin 的账号看板永远转圈。统一用 roleOf。

### P1-5 小红书链接校验可被绕过

`lib/kos_store.py:542-556`:正则只要求域名字符串出现在 URL 任意位置,`https://evil.com/xiaohongshu.com`、`https://xhslink.com.evil.com/` 都能过,入库后呈现给管理员点击 = 钓鱼入口。**修复**:`urlsplit` 后校验 hostname 精确匹配/合法子域。

### P1-6 前端仍全量注入 `_user`,与 v3.2 安全升级目标矛盾

8 处页面手工塞 `_user`,`index.html:1200-1216` 还有全局 fetch 补丁自动注入(app.js 已打过一次补丁,index 又包一层)。后端 v3.2 已不信任 `_user`,这些是残留物,但只要后端还有任何一处读它做记账就是伪造面。**修复**:逐接口确认后全删。

### P1-7 并发与性能

- 每次调用开新 SQLite 连接 + PRAGMA,`list_tasks` 对每个任务再各开连接查 capacity(1+3N 连接风暴);`list_libraries` 每次打开页面对每个库全量 COS 对账(`api/kos-admin.py:181-190`),很慢。→ 抽 `lib/db.py`:threading.local 连接复用 + 统一 PRAGMA;capacity 批量化;COS 对账改按需。
- AI 调用无全局并发上限:cover-generate 每请求 1 主线程 + 3 出图线程,等待可达 190s,突发可耗尽线程。→ 全局信号量限制在途 AI 请求数。
- `sys.path.insert` 在请求路径里反复执行(generate/qa/library 多处),path 无限增长。→ 统一模块顶部判重插入。
- `dev_server.py:176`:handler 已部分写出响应后再抛异常,兜底会往同一 wfile 追加 JSON,产生畸形报文。→ 判断已写出则只记日志。
- `lib/kos_store.py:306-317`:组合兜底穷举在竖版 100+ 张时一次性物化数百万组合,可吃数 GB 内存。→ 改迭代器水塘抽样。
- `lib/image_compose.py`:Image.open 从不 close,fd 缓慢泄漏。→ `with Image.open(...)`。
- `lib/ratelimit.py:62-90`:每日配额先 SELECT 再 UPSERT 有小竞态;IP 取 client_address,Nginx 反代下全员共享 127.0.0.1 一个桶。→ UPSERT RETURNING;信任本机反代时取 X-Forwarded-For。

## P2 — 质量/可维护性(迭代顺手做)

1. **抽 `lib/http_base.py` 基类**:15 个 api 文件各自复制 CORS 四件套/_json/JSON 解析/鉴权样板,近 200 行重复。抽基类可一次性顺带修掉 P0-5(错误泄漏)和 CORS 收敛,收益最高的一项重构。
2. **抽 `lib/cos.py` / `lib/deepseek.py`**:COS client 构造 4 份、DeepSeek 调用 3 份。
3. **前端收口 app.js**(可核减 600-700 行,约 10%):escapeHtml 5 份、会话校验样板 4 份手写、登录表单 2 份、剪贴板 3 套、COS 直传循环 2 份 → 收进 App.copyText / App.uploadToCos;老四页(portal/index/qa/admin)迁移到 App.requireSession。
4. **删死代码**:index.html:1269-1442 残留整套旧管理面板(170 行,还硬编码 Upstash 部署细节);api/generate.py:35-46、cover-fields.py:31-42、login.py:34-36 的废弃 load_products/load_users;kv_store 的 get_counter;index.html 约 150 行死 CSS。
5. **拆前端文件(不引构建)**:index.html 1780 行全内联无法缓存 → 每页拆 `/assets/js/<page>.js`;Chart.js 从 jsdelivr 改自托管(大陆环境 CDN 挂 = 看板全白)。
6. **哈希降级明文**:`lib/users_store.py:119-125` idhash 失败静默存明文身份证后6位,应显式报错(P0-4 修复后此分支应删)。
7. **备份增强**:`scripts/backup_db.py` 用 backup API 是对的,但缺 tmp+rename 原子写、integrity_check、离机副本(rsync/COS)。
8. **KOS 上传预签名**:限定 Content-Type 与大小;brand/product 进 COS key 前做字符白名单。
9. **token 无吊销机制**:停用员工凭手里 token 在 24h 内仍可用 generate/qa/下载。→ 非管理接口也做一次轻量 active 校验(连接复用后成本极低)。
10. **杂项**:.gitignore 第 9 行两条规则粘连(`DOUBAO_SETUP产品库/`);qa.py prompt 先禁"依据"又要求"依据"再正则删掉,自相矛盾;kos_store 的 org 参数名存实亡(leaderboard 的 org 参数从未使用);多处 docstring 仍讲 Vercel Serverless 与现状不符;kv_store 四合一职责应拆分;LIKE 查询未转义 %_。
